# 校园活动报名系统 - 课设答辩要点总结

## 一、项目介绍（1-2分钟）

### 1.1 项目背景
- **项目名称**：基于Web的校园活动报名系统
- **面向场景**：高校校园活动管理，解决传统人工报名方式效率低、管理难的问题
- **目标用户**：系统管理员、社团/院系负责人、普通在校学生

### 1.2 核心功能
本系统实现了校园活动从**发布 → 审核 → 报名 → 审核 → 统计导出**的全流程数字化管理：

| 模块 | 功能点 |
|------|--------|
| 用户权限模块 | 基于RBAC的三角色权限控制（管理员/负责人/学生） |
| 活动发布管理 | 活动CRUD、多状态流转（草稿→待审核→已发布→报名中→已结束） |
| 在线报名模块 | 活动搜索筛选、一键报名、自动校验（时间/人数/重复报名） |
| 报名审核模块 | 负责人审核报名申请（通过/驳回+备注原因） |
| 数据统计导出 | 报名人数统计、通过率统计、导出CSV名单 |
| 公告通知模块 | 公告发布置顶、浏览量统计 |

## 三、核心架构讲解（2-3分钟）

### 3.1 系统架构图

```
┌─────────────────────────────────────────┐
│               前端 (Vue 3)               │
│   Element Plus UI + Axios + Vue Router  │
│              端口: 8081                  │
└──────────────┬──────────────────────────┘
               │ HTTP/RESTful API
               │ JWT Bearer Token
┌──────────────▼──────────────────────────┐
│              后端 (Spring Boot)          │
│  ┌─────────┐ ┌─────────┐ ┌──────────┐  │
│  │Controller│ │ Service │ │  Mapper  │  │
│  │  接口层  │ │ 业务层  │ │ 数据层   │  │
│  └─────────┘ └─────────┘ └──────────┘  │
│         MyBatis-Plus ORM框架            │
│              端口: 8080                  │
└──────────────┬──────────────────────────┘
               │ SQL
┌──────────────▼──────────────────────────┐
│           MySQL 8.0 数据库              │
│   user / role / activity / registration │
│   完整外键约束 + 索引优化               │
└─────────────────────────────────────────┘
```

### 3.2 分层架构说明

```
Controller层 → 接收HTTP请求，参数校验，返回统一Result
     ↓
Service层    → 业务逻辑处理，事务管理，权限校验
     ↓
Mapper层     → 数据访问，SQL执行（MyBatis-Plus）
     ↓
Entity层     → 数据模型，与数据库表一一对应
```

### 3.3 权限控制设计（重点加分项）

采用 **RBAC（Role-Based Access Control）** 模型：

```
用户(User) ←→ 用户角色关联(UserRole) ←→ 角色(Role)
                                    
管理员(admin):     所有接口权限
负责人(organizer): 活动发布、报名审核、名单导出
学生(student):     活动浏览、在线报名

实现方式：JWT Token + ThreadLocal + 拦截器
- 登录时生成JWT Token，包含用户ID和角色编码
- 每次请求携带Token，拦截器解析并验证
- ThreadLocal存储当前用户，Service层进行权限校验
```

## 五、遇到的问题及解决方案

| 问题 | 解决方案 |
|------|----------|
| 跨域问题（CORS） | 后端配置CorsFilter，允许前端8081端口访问 |
| JWT Token过期处理 | 响应拦截器捕获401状态码，自动跳转登录页 |
| 数据库时间时区错误 | JDBC URL添加 `serverTimezone=Asia/Shanghai` |
| 密码明文存储安全问题 | 使用BCryptPasswordEncoder加密存储 |
| 并发报名人数超上限 | 数据库乐观锁（SQL条件判断），超限返回失败 |
| 前端路由权限控制 | 路由元信息 `meta.roles` + 路由守卫判断 |

## 七、答辩常见问题及回答

**Q1: 为什么选择Spring Boot而不是Spring MVC？**
> A: Spring Boot提供自动配置、内嵌Tomcat、起步依赖，开发效率更高，是目前Java Web开发的主流选择。

**Q2: JWT和Session的区别？为什么选JWT？**
> A: JWT是无状态的，服务端不需要存储会话信息，适合前后端分离和分布式部署。Session需要服务端存储，不适合分布式。

**Q3: 如何防止报名人数超过上限？**
> A: 使用数据库乐观锁，通过SQL条件 `current_participants < max_participants` 判断，并发时只有一个成功。

**Q4: RBAC权限控制怎么实现的？**
> A: 通过用户-角色关联表实现，JWT Token携带角色信息，拦截器解析后在Service层进行权限校验。

**Q5: 前后端怎么交互的？**
> A: 通过Axios发送HTTP请求，使用JWT Token放在请求头Authorization中进行身份认证，后端返回JSON数据。
