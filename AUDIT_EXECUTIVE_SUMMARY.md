## ⚡ 最严重的 3 个问题

### 1️⃣ [CRITICAL] 并发报名超卖漏洞 (D3-001)
- **风险**: 用户可在活动满员后仍报名成功
- **原因**: 竞速条件 - 检查和修改之间无锁
- **影响**: 数据完整性破坏、业务逻辑崩溃
- **修复时间**: 2-4 小时
- **具体位置**: `backend/mapper/ActivityMapper.java` 49-50 行

**示例**:
```
活动限制: 1人
同时2人报名 → 两个请求都通过检查 → 最终2人报名成功 ❌
```

**推荐方案**: 使用数据库行级锁 (SELECT ... FOR UPDATE)

---

### 2️⃣ [CRITICAL] 审核驳回状态错误 (D2-001)
- **风险**: 被驳回的活动无法编辑重新提交
- **原因**: 驳回后状态设为 1(待审核) 而非 0(草稿)
- **影响**: 组织者体验差、活动发布流程中断
- **修复时间**: < 1 小时
- **具体位置**: `backend/service/impl/ActivityServiceImpl.java` 180-188 行

**修改前后**:
```java
// 之前（错误）
if (status == 2) activity.setStatus(2);  // 通过
else activity.setStatus(1);  // 驳回 ❌ 应为 0

// 之后（正确）
if (status == 2) activity.setStatus(2);  // 通过
else activity.setStatus(0);  // 驳回 ✓ 回到草稿
```

---

### 3️⃣ [CRITICAL] 统计数据空指针异常 (D6-004)
- **风险**: 服务端崩溃，用户无法查看活动统计
- **原因**: 未检查聚合查询的 null 结果
- **影响**: 拒绝服务 (DoS) 风险
- **修复时间**: < 1 小时
- **具体位置**: `backend/service/impl/ActivityServiceImpl.java` 244-246 行

**修复方案**:
```java
// 之前（会报错）
long approved = Long.parseLong(result.get("approvedCount").toString());  // NPE!

// 之后（安全）
long approved = result.get("approvedCount") != null ? 
                Long.parseLong(result.get("approvedCount").toString()) : 0L;
```
